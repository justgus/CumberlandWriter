## ER-0008: Time-Based Timeline System

### Decision 1: Calendar System Data Model

**Question:** Should calendar systems be stored as JSON in Rules cards or as a dedicated SwiftData model?

**Decision:** **Dedicated SwiftData Model**

**Rationale:**
- **Queryability:** SwiftData models support predicates and queries; JSON fields do not
- **Relationships:** Can establish typed relationships (Timeline → CalendarSystem)
- **CloudKit Sync:** Native support for complex types
- **Validation:** Model-level validation enforced by SwiftData
- **Performance:** Indexed queries faster than JSON parsing
- **Type Safety:** Compile-time type checking vs runtime JSON parsing

**Implementation:**
```swift
@Model
final class CalendarSystem {
    var id: UUID
    var name: String
    var divisions: [TimeDivision]
    var createdAt: Date

    // Relationship to timelines using this calendar
    @Relationship(inverse: \Timeline.calendarSystem)
    var timelines: [Card]?

    init(name: String, divisions: [TimeDivision]) {
        self.id = UUID()
        self.name = name
        self.divisions = divisions
        self.createdAt = Date()
    }
}

struct TimeDivision: Codable {
    var name: String
    var pluralName: String
    var length: Int // number of subdivisions
    var isVariable: Bool // e.g., months have 28-31 days
}
```

**Trade-offs:**
- ✅ Better performance and type safety
- ✅ Easier to query and filter
- ❌ Additional model to manage
- ❌ Requires schema migration

### Decision 2: Epoch Storage

**Question:** Should epoch be a property on Timeline or a separate model?

**Decision:** **Property on Timeline Card**

**Rationale:**
- **Simplicity:** One epoch per timeline; no need for separate model
- **Cohesion:** Epoch is conceptually part of the timeline definition
- **Performance:** No additional database join required
- **Migration:** Easier to add properties than new models
- **User Experience:** Simpler UI (edit timeline → set epoch inline)

**Implementation:**
```swift
// Add to Card model (for Timeline kind)
```

### Decision 3: Timeline Zoom Behavior

**Question:** Should zoom be linear, logarithmic, or adaptive?

**Decision:** **Adaptive Zoom (Automatic based on data range)**

**Rationale:**
- **Flexibility:** Works well for short timelines (hours) and long timelines (millennia)
- **User Experience:** Automatically selects appropriate zoom level
- **Smart Defaults:** Hour → Day → Week → Month → Year → Decade → Century
- **User Override:** Can still implement manual zoom controls
- **Similar to Best Practices:** Gantt chart tools use adaptive zoom

**Implementation:**
```swift
enum TimelineZoomLevel: CaseIterable {
    case hour, day, week, month, year, decade, century, millennium

    var duration: TimeInterval {
        switch self {
        case .hour: return 3600
        case .day: return 86400
        case .week: return 604800
        case .month: return 2592000 // ~30 days
        case .year: return 31536000
        case .decade: return 315360000
        case .century: return 3153600000
        case .millennium: return 31536000000
        }
    }

    static func automatic(for timeRange: TimeInterval) -> TimelineZoomLevel {
        // Select zoom level that shows 5-20 major divisions
        let targetDivisions = 10.0
        for level in Self.allCases {
            if timeRange / level.duration < targetDivisions * 2 {
                return level
            }
        }
        return .millennium
    }
}
```

**Trade-offs:**
- ✅ Works for all timeline scales
- ✅ Intuitive default behavior
- ❌ Slightly more complex implementation
- ❌ May require fine-tuning of thresholds

## ER-0009: AI Image Generation

### Decision 4: Attribution Display

**Question:** Should attribution be a subtle badge or visible overlay?

**Decision:** **Subtle Badge with Tap-to-Expand**

**Rationale:**
- **User Preference:** Most users prefer subtle UI
- **Compliance:** Still meets legal requirements (visible attribution)
- **Flexibility:** User can enable "always show overlay" in settings
- **Best Practices:** Similar to stock photo watermarks
- **Non-Intrusive:** Doesn't distract from image viewing

**Implementation:**
```swift
// Settings
@AppStorage("aiAttribution_alwaysShowOverlay")
var alwaysShowOverlay: Bool = false

// UI
struct CardImageView: View {
    let card: Card
    @AppStorage("aiAttribution_alwaysShowOverlay") var showOverlay = false

    var body: some View {
        ZStack(alignment: .bottomTrailing) {
            Image(data: card.imageData)

            if card.imageGeneratedByAI == true {
                if showOverlay {
                    AttributionOverlay(card: card)
                } else {
                    AttributionBadge(card: card)
                        .onTapGesture {
                            // Show full attribution info panel
                        }
                }
            }
        }
    }
}
```

**Trade-offs:**
- ✅ Clean visual appearance
- ✅ User control over prominence
- ✅ Legal compliance maintained
- ❌ Badge might be overlooked by some users

### Decision 5: Regeneration History

**Question:** Store all versions or latest only?

**Decision:** **Configurable Limit (Default: Last 5 Versions)**

**Rationale:**
- **Storage Management:** Prevents unlimited storage growth
- **User Control:** Power users can increase limit
- **Practical Balance:** 5 versions sufficient for most use cases
- **Performance:** Limit prevents UI slowdown from huge version lists
- **MVP Scope:** Can expand to full version control later

**Implementation:**
```swift
@Model
final class Card {
    // ... existing properties ...

    // ER-0009: Image generation metadata
    var imageGeneratedByAI: Bool?
    var imageAIProvider: String?
    var imageAIPrompt: String?
    var imageAIGeneratedAt: Date?

    // ER-0009: Regeneration history (last N versions)
    @Attribute(.externalStorage)
    var imageHistoryData: Data? // Array of previous images (JPEG)
    var imageHistoryMetadata: [ImageHistoryEntry]?
}

struct ImageHistoryEntry: Codable {
    var timestamp: Date
    var provider: String
    var prompt: String
    var version: Int
}

// Settings
@AppStorage("aiGeneration_historyLimit")
var historyLimit: Int = 5 // Configurable
```

**Trade-offs:**
- ✅ Manageable storage impact
- ✅ User flexibility
- ✅ Sufficient for practical use
- ❌ Not full version control (acceptable for MVP)

### Decision 6: Auto-Generation Trigger

**Question:** Should images auto-generate on save or on-demand only?

**Decision:** **Optional Setting, Off by Default**

**Rationale:**
- **User Control:** Users opt-in to automation
- **API Costs:** Prevents unexpected API charges (OpenAI)
- **User Intent:** Respects when user explicitly doesn't want images
- **Bandwidth:** Prevents unwanted network usage
- **Power Users:** Those who want it can enable it

**Implementation:**
```swift
// Settings
@AppStorage("aiGeneration_autoGenerate")
var autoGenerateImages: Bool = false

@AppStorage("aiGeneration_autoGenerateMinWords")
var autoGenerateMinWords: Int = 50

// Trigger logic
extension Card {
    func shouldAutoGenerateImage() -> Bool {
        guard UserDefaults.standard.bool(forKey: "aiGeneration_autoGenerate") else {
            return false
        }

        // Check conditions
        let hasNoImage = originalImageData == nil
        let hasDescription = !(detailedText ?? "").isEmpty
        let wordCount = (detailedText ?? "").split(separator: " ").count
        let meetsMinimum = wordCount >= UserDefaults.standard.integer(forKey: "aiGeneration_autoGenerateMinWords")

        return hasNoImage && hasDescription && meetsMinimum
    }
}
```

**Trade-offs:**
- ✅ User control and consent
- ✅ No surprise costs
- ✅ Explicit intent respected
- ❌ Slightly more setup for users who want automation

## ER-0010: AI Content Analysis

### Decision 7: Analysis Triggers

**Question:** Manual only or auto-analyze option?

**Decision:** **Manual Only (MVP), Auto-Analyze as Future Enhancement**

**Rationale:**
- **User Control:** User decides when to analyze
- **API Costs:** Prevents runaway API usage
- **Privacy:** User explicitly consents to sending text to AI
- **Performance:** Doesn't slow down typing/editing
- **MVP Scope:** Simpler implementation, can add auto-analyze later
- **User Expectation:** "Analyze" button makes intent clear

**Implementation:**
```swift
struct CardEditorView: View {
    @State private var showingSuggestions = false

    var body: some View {
        Form {
            Section("Description") {
                TextEditor(text: $card.detailedText)

                Button {
                    analyzeDescription()
                } label: {
                    Label("Analyze", systemImage: "sparkles")
                }
                .disabled(card.detailedText.split(separator: " ").count < 25)
            }
        }
        .sheet(isPresented: $showingSuggestions) {
            SuggestionReviewView(suggestions: suggestions)
        }
    }
}
```

**Trade-offs:**
- ✅ Clear user intent
- ✅ No surprise API calls
- ✅ Simpler MVP implementation
- ❌ Requires manual trigger (acceptable for MVP)

**Future Enhancement:** Add setting for auto-analyze with configurable delay (e.g., 5 seconds after typing stops)

### Decision 8: Suggestion Persistence

**Question:** Store suggestions for later or discard on close?

**Decision:** **Discard on Close (MVP), Persistent Queue as Future Enhancement**

**Rationale:**
- **Simplicity:** No additional data model required
- **User Flow:** Review suggestions immediately (common case)
- **MVP Scope:** Simpler implementation
- **Storage:** Doesn't accumulate stale suggestions
- **User Expectation:** Modal review pattern is familiar

**Implementation:**

### Decision 9: Learning from User Patterns

**Question:** Should system track user accept/reject patterns?

**Decision:** **Yes, Anonymized and Local-Only**

**Rationale:**
- **Improved Accuracy:** Learn user preferences over time
- **Privacy:** No data sent to external servers
- **Personalization:** Adjust confidence thresholds per user
- **User Benefit:** Fewer false positives over time
- **Ethical:** Transparent, local-only learning

**Implementation:**
```swift
@Model
final class SuggestionFeedback {
    var id: UUID
    var entityType: String // "Location", "Character", etc.
    var confidence: Double
    var wasAccepted: Bool
    var timestamp: Date

    // No actual entity names stored (privacy)

    init(entityType: String, confidence: Double, wasAccepted: Bool) {
        self.id = UUID()
        self.entityType = entityType
        self.confidence = confidence
        self.wasAccepted = wasAccepted
        self.timestamp = Date()
    }
}

// Use feedback to adjust thresholds
class SuggestionEngine {
    func adjustedConfidenceThreshold(for entityType: String) -> Double {
        // Query feedback history
        // If user consistently rejects high-confidence suggestions,
        // raise the threshold for that entity type

        let feedback = queryFeedback(entityType: entityType)
        let baseThreshold = 0.70

        // Calculate acceptance rate
        let recentFeedback = feedback.suffix(20)
        let acceptanceRate = recentFeedback.filter(\.wasAccepted).count / max(recentFeedback.count, 1)

        // Adjust threshold (if acceptance < 60%, raise threshold)
        if acceptanceRate < 0.6 {
            return min(baseThreshold + 0.1, 0.9)
        }

        return baseThreshold
    }
}
```

**Trade-offs:**
- ✅ Improves over time
- ✅ Privacy-preserving
- ✅ User-specific personalization
- ❌ Additional data model (acceptable - small footprint)

## Shared AI Infrastructure

### Decision 10: AI Provider Architecture

**Decision:** **Protocol-Based Provider System with Apple Intelligence Default**

**Rationale:**
- **Extensibility:** Easy to add new providers (ChatGPT, Claude, future)
- **Testability:** Mock providers for unit tests
- **Separation of Concerns:** Provider logic isolated
- **Platform Support:** Apple Intelligence available on latest OS versions
- **Fallback Strategy:** Clear fallback hierarchy

**Implementation:**
```swift
protocol AIProviderProtocol {
    var name: String { get }
    var isAvailable: Bool { get }
    var requiresAPIKey: Bool { get }

    // ER-0009: Image generation
    func generateImage(prompt: String) async throws -> Data

    // ER-0010: Content analysis
    func analyzeText(_ text: String, for task: AnalysisTask) async throws -> AnalysisResult
}

enum AnalysisTask {
    case entityExtraction
    case relationshipInference
    case calendarExtraction
}

struct AnalysisResult {
    var entities: [Entity]?
    var relationships: [Relationship]?
    var calendar: CalendarStructure?
}

// Concrete providers
class AppleIntelligenceProvider: AIProviderProtocol {
    var name: String { "Apple Intelligence" }
    var isAvailable: Bool {
        #if os(iOS)
        return ProcessInfo.processInfo.operatingSystemVersion.majorVersion >= 18
        #elseif os(macOS)
        return ProcessInfo.processInfo.operatingSystemVersion.majorVersion >= 15
        #else
        return false
        #endif
    }
    var requiresAPIKey: Bool { false }

    // Implementation uses ImagePlayground framework
}

class OpenAIProvider: AIProviderProtocol {
    var name: String { "OpenAI (ChatGPT)" }
    var isAvailable: Bool { hasValidAPIKey() }
    var requiresAPIKey: Bool { true }

    // Implementation uses OpenAI API
}

// Provider registry
class AIProviderRegistry {
    static let shared = AIProviderRegistry()

    private let providers: [AIProviderProtocol] = [
        AppleIntelligenceProvider(),
        OpenAIProvider()
    ]

    func defaultProvider() -> AIProviderProtocol? {
        providers.first(where: \.isAvailable)
    }

    func provider(named name: String) -> AIProviderProtocol? {
        providers.first { $0.name == name }
    }
}
```

**Trade-offs:**
- ✅ Clean abstraction
- ✅ Easy to extend
- ✅ Testable
- ❌ Requires protocol design upfront (acceptable - well-defined)

## Schema Migration Strategy

### Decision 11: Schema Versioning

**Decision:** **Incremental Schema Versions (V6, V7, V8 as needed)**

**Rationale:**
- **Risk Management:** Smaller migrations easier to test and rollback
- **Parallel Development:** Different ERs can have different schema versions if needed
- **CloudKit Compatibility:** Gradual schema evolution supported
- **Rollback Safety:** Easier to revert smaller changes

**Plan:**
- **AppSchemaV6:** Calendar System model, epoch properties
- **AppSchemaV7:** AI metadata properties (imageGeneratedByAI, etc.)
- **AppSchemaV8:** Suggestion feedback model (if needed in V6/V7, combine)

**Implementation:**
```swift
// In Migrations.swift

enum AppSchemaV6: VersionedSchema {
    static var versionIdentifier = Schema.Version(6, 0, 0)
    static var models: [any PersistentModel.Type] {
        [Card.self, CalendarSystem.self, StoryStructure.self, /* ... */]
    }
}

enum AppSchemaV7: VersionedSchema {
    static var versionIdentifier = Schema.Version(7, 0, 0)
    static var models: [any PersistentModel.Type] {
        [Card.self, CalendarSystem.self, StoryStructure.self, /* ... */]
    }
}

// Migration stages
extension AppMigrations {
    static var schemas: [any VersionedSchema.Type] {
        [AppSchemaV1.self, AppSchemaV2.self, AppSchemaV3.self,
         AppSchemaV4.self, AppSchemaV5.self,
         AppSchemaV6.self, AppSchemaV7.self]
    }

    static var stages: [MigrationStage] {
        [
            // ... existing migrations ...
            MigrationStage.lightweight(fromVersion: AppSchemaV5.self, toVersion: AppSchemaV6.self),
            MigrationStage.lightweight(fromVersion: AppSchemaV6.self, toVersion: AppSchemaV7.self)
        ]
    }
}
```

**Decision:** **AppStorage-Based Feature Flags for Beta/Rollout**

**Rationale:**

- **Gradual Rollout:** Can enable for subset of users

- ✅ Quick rollback capability
- ✅ Flexible deployment
- ❌ Additional conditional logic (cleaned up after stable release)
